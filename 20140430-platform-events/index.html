Awesome Inside!

We've snuck an extra special treat into our latest release (version 1.5) of the DreamFactory Services Platform™. This new morsel of awesome is the new platform events and scripting system. These two new built-in services (/rest/system/event and /rest/system/script) allows you, dear developer, the ability to synchronously listen for, and act upon, system events in real-time. That's a lot to digest in one sentence so a little background is in order.

What Good Are Events?

Since the dawn of humankind, events have been a part of our lives. Holidays, birthdays, parties, coronation ceremonies, you name it; unless you're some kind of hermit, you've probably attended an event. The problem with events is that you always need a reminder. As technology has progressed, we can get notified of upcoming events from our devices and social and organizational networking systems. It's only natural that someone had the idea to bring this concept to software systems.

Having the system tell you when something is about to, or has, occurred is a blessing in disguise. Once, long ago dear reader, we were forced to poll for system changes. But we endured until our platforms embraced events. These nuggets of system information, jam-packed full of goodies, are sent out by our systems, allow you to build your applications more easily. What's that? Easy you say? Pssshaw!

No, really, it is easier (if we did a good job implementing our event system ;) ). Consider a typical client-side developer (named Pat). Pat knows javascript and HTML5 hands-down. But needs a specialized service that checks a remote server for additional authentication information. Perhaps for two-factor checks. But to build a service like that -- and incorporate it into the DreamFactory Services Platform™ -- can be a daunting task to they who are not one with PHP. This is where events rock.

All Pat really needs to know is when someone is about to log in to the system. At that point the app can make the remote call and either allow or deny the operation. The particular one that would be useful in our case is session.login. Pat's app registers as a listener for this event and when it's fired by the server, the URL registered as an event listener, will be called via an HTTP POST. This isn't the only way apps can consume events. You may also create PHP event handlers and server-side javascript (more on this later).

How Does It Work?

The DreamFactory Services Platform™ is architected in such a way that we can hook, inject, and augment REST API calls. This feature is implemented and demonstrated in the preProcess() and postProcess() methods of our base service and resource classes. We enhanced this system to dispatch events when REST API calls are processed. One of our major design goals was that the platform be merely a dispatcher of events, and not define them. Yes, the actions performed by the server trigger the events, but the names of these events are virtually ignored by the server.
